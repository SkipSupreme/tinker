# Accessibility checklist

Every interactive demo on this site must pass this checklist before merging.
This is the explicit, enumerated version of the promise the home page makes:
**svelte-mafs is keyboard-accessible from day one.**

If you're adding a Wave B example page, copy the checklist below into the PR
description and tick each box. If something doesn't apply, strike it through
with a one-line reason.

---

## Keyboard

- [ ] Every interactive element reaches focus by Tab, in a sensible order.
      No `tabindex` greater than 0 unless there's a compelling reason.
- [ ] `Shift+Tab` reverses the order without surprises.
- [ ] `Enter` / `Space` activate buttons and toggles.
- [ ] `Esc` dismisses any dialogs or modal previews.
- [ ] Movable points move by:
  - [ ] Arrow keys (1 user-unit per press, or the configured step).
  - [ ] `Shift+Arrow` for 10× step (coarse move).
  - [ ] `Home` / `End` for axis-aligned extremes where meaningful.
- [ ] The demo never traps focus. Users can Tab out to the page nav at any time.
- [ ] Keyboard-initiated drags emit the same `onDragStart` / `onDragEnd`
      sequence as pointer drags — no divergent paths.

## Focus indication

- [ ] Focus rings use `:focus-visible` (pointer click doesn't show a ring;
      keyboard focus does).
- [ ] Rings are drawn against `--site-focus` / `--mafs-focus` with at least
      3:1 contrast ratio against both light and dark backgrounds.
- [ ] Ring isn't clipped by `overflow: hidden` on SVG containers
      (`outline-offset` rescues this; test it).

## Screen reader

- [ ] Every `<Mafs>` root includes an `aria-label` describing what the
      figure shows.
- [ ] Purely decorative SVGs have `aria-hidden="true"` and no text content.
- [ ] `<MovablePoint>` has `role="slider"` with `aria-valuenow-x` /
      `aria-valuenow-y` (and `aria-valuetext` for formatted announcements
      like `"x = 1.5, y = −2.3"`).
- [ ] Value changes announce via a single `aria-live="polite"` region at the
      `<Mafs>` root. Polite, not assertive — we don't interrupt.
- [ ] The region announces only on `dragend` / `keyup`, not on every
      intermediate frame (avoids spam).
- [ ] `<Plot>` curves have a text-equivalent summary nearby (e.g. "Sine
      wave from −2π to 2π").
- [ ] Axis labels are rendered as real `<text>` or KaTeX DOM — never as
      images.

## Color + contrast

- [ ] Foreground vs background meets WCAG AA (4.5:1 normal text, 3:1 large).
- [ ] Axis lines and grid meet 3:1 against the canvas background.
- [ ] Information is never carried by color alone: pair color with shape,
      stroke pattern, or a label.
- [ ] All theme tokens pass the check on both light and dark modes.

## Motion

- [ ] `@media (prefers-reduced-motion: reduce)` disables any animated
      pan / zoom easing — changes are immediate.
- [ ] Demos that auto-animate (rare) have a pause button and respect
      reduced-motion.

## Pointer / touch

- [ ] Hit targets are at least 44 × 44 CSS px (iOS HIG minimum).
- [ ] No `dblclick` handlers — they're unreliable on touch. Use single tap
      plus a secondary UI.
- [ ] Long-press does nothing surprising (selection is fine).
- [ ] Pointer capture is released on `pointercancel` (don't leave drag
      stuck if the user rotates their device mid-drag).

## Structure + meta

- [ ] Page has a single `<h1>`. Demo titles are `<h2>` or below.
- [ ] `<title>` is descriptive and unique per route.
- [ ] Landmarks (`<nav>`, `<main>`, `<footer>`) are present exactly once.
- [ ] Skip-to-content link is visible on focus (handled by `Base.astro`).

## Hydration

- [ ] The demo renders a meaningful static SVG before hydration — a screen
      reader reaching the page mid-load still sees the shape, not an empty
      placeholder.
- [ ] Islands use `client:visible` or `client:idle`. `client:load` only if
      the demo is above the fold and needs instant interactivity.
- [ ] SSR-rendered labels match client-rendered labels exactly
      (no hydration-mismatch warnings in the console).

## Test evidence

For each example PR, record:

- Screen reader: VoiceOver (macOS) + NVDA (Windows). Describe what's
  announced when tabbing into the demo and activating it.
- Keyboard: record a one-minute keyboard-only walk-through (video or
  GIF) and attach to the PR.
- Contrast: link to WebAIM contrast checker results for the main color
  pairings.
- Lighthouse a11y score ≥ 95 on the deployed preview.

---

## Out of scope for this checklist

These are desirable but not blocking:

- Custom reduced-contrast theme.
- Braille rendering of LaTeX labels (KaTeX's a11y tree is reasonable).
- High-contrast Windows theme compatibility (monitor separately).

If you hit a case the checklist doesn't cover, add a row here in the same
PR. The goal is for this file to be the single source of truth.
